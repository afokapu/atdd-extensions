# W11 classification — Tester: security + artifact + telemetry + isolation

Source (read-only): `/Users/alecfokapu/Github/atdd/main/src/atdd/tester/conventions/`

Method: for every rule_id in the assigned conventions, classify **AGNOSTIC-CONSUMER**
(the obligation applies to any consumer repo's code/tests → MIRROR into the
Convex/Vitest stack) or **ATDD-INTERNAL / SUBSTRATE-SPEC / PLANNER-OWNED** (governs
only the ATDD toolkit's own substrate or a `SPEC-*` internal validator id →
DOCUMENT WHY-NOT, no mirror). Conservative rule: wording that targets a *consumer*'s
code/tests, or that has per-stack renderings, is AGNOSTIC.

---

## 1. `tester/conventions/security.convention.yaml`

This file has two distinct layers: (a) a bank of `SPEC-TESTER-SEC-*` validator specs
that validate the ATDD **contract-schema substrate** (OpenAPI-style operations with
`security[]`, `scheme_header_mapping`, `threat_modeling` → `feature.yaml`), and
(b) two `#394` structured rules that govern a *consumer's test suite*.

| rule_id | class | evidence |
|---|---|---|
| `SPEC-TESTER-SEC-0001` (`test_secured_operations_have_required_headers`) | **ATDD-INTERNAL / SUBSTRATE-SPEC** | `name: "Security Convention" … "Security validation rules for contract schemas"`; the test operates on `security[]`/`scheme_header_mapping` of the toolkit's own contract schema. `SPEC-*` id ⇒ internal spec, per method. |
| `SPEC-TESTER-SEC-0002` (`test_operations_have_explicit_security`) | **ATDD-INTERNAL / SUBSTRATE-SPEC** | `"All API operations should declare their security requirements explicitly. Use security: [] …"` — governs the contract-schema `security` field format, not consumer code. `SPEC-*` id. |
| `SPEC-TESTER-SEC-0003` (`test_secured_operations_have_security_acceptance`) | **ATDD-INTERNAL / SUBSTRATE-SPEC** | `"acceptance criteria using the SEC (Security) or RLS (Row-Level Security) harness"` + `urn_format: "acc:{wagon}:{WMBT}-{HARNESS}-{NNN}…"` — a plan-layer acceptance-linkage check over ATDD URNs. `SPEC-*` id, planner-owned. |
| `SPEC-TESTER-SEC-0004` (`test_error_responses_have_no_sensitive_fields`) | **ATDD-INTERNAL / SUBSTRATE-SPEC** | `"Error responses (4xx/5xx) should not contain fields …"` — inspects the contract schema's error-response `properties`, `mode: "warning only"`; a substrate schema linter. `SPEC-*` id. (The consumer-code analogue — error responses not leaking secrets — is `coder.convex.error-response-*`, already built.) |
| `tester.security.auth` (`TESTER-SECURITY-AUTH-001`) | **AGNOSTIC-CONSUMER → MIRROR** | `description: "Security-sensitive endpoints are covered by tests asserting auth and authorization"` — a `#394` structured rule whose obligation is on *any* consumer's test suite. `disposition: documentation-only`. → `tester.convex.security-auth`. |
| `tester.security.input` (`TESTER-SECURITY-INPUT-001`) | **AGNOSTIC-CONSUMER → MIRROR** | `description: "Input-validation tests assert rejection of malformed and adversarial payloads"` — obligation on a consumer's test suite. `disposition: documentation-only`. → `tester.convex.security-input`. |

**Why the SPEC-* half is not mirrored.** Every `SPEC-TESTER-SEC-000x` targets the
ATDD *contract schema* substrate (`security[]`, `scheme_header_mapping`, error-response
`properties`) — the toolkit's own artifact format — not a consumer's Convex/TS code.
The method reserves `SPEC-*` ids for DOCUMENT-WHY-NOT. The genuinely consumer-facing
obligations in this file are the two `#394` structured rules, both mirrored below.

## 4. `tester/conventions/artifact.convention.yaml`

| section | class | evidence |
|---|---|---|
| **entire file** (`naming`, `versioning`, `organization`, `api_mapping`, `ownership`, `metadata`, `validation`, `artifact_urns`, `wagon_artifacts`, `contract_artifacts`, `demo_mode`) | **ATDD-INTERNAL / SUBSTRATE-SPEC** | `description: "Artifact-centric contract system using theme-based hierarchical taxonomy"`; `physical_pattern: "contracts/{theme}/{segments}/{aspect}[/{variant}].schema.json"`; `ownership.producer: "wagon that declares artifact in produce[]"`. The whole convention describes how the ATDD **contract registry** (`contracts/…schema.json`) is named/versioned/owned by *wagons* — a planner/substrate artifact format. It carries **no `rules:` block and no structured rule_id**, so there is no enforceable consumer obligation to render. `commons:ux:foundations`, `mechanic:decision.choice`, etc. are artifact-taxonomy identifiers of that registry, not consumer code targets. |

**Why-not (artifact):** DOCUMENT-WHY-NOT in full. This is a naming/organization spec for
the toolkit's artifact contracts (owned by wagons at plan time). No per-stack variant,
no "a consumer MUST …" wording, no structured rule id ⇒ nothing to mirror into a
Convex/TS consumer. Mirroring it would invent a consumer obligation the core never
states.
