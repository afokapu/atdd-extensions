## AGNOSTIC-CONSUMER → MIRROR (W7 BUILD — genuine gaps)

New family detector **`frontend.workspace.runtime/implementations/vite_design_hierarchy_detector/`**
(one FAMILY validator, Core pattern — three cohesive `emits_rule_ids`). Frontend/
Vite is the canonical home of the `design_system/{tokens,primitives,components,templates}/`
hierarchy the convention describes (`location.path: "lib/design_system/"`;
frg-app ground truth is `apps/game/src/`, a Vite app).

### 1. `coder.design.tokens-are-pure-values` (DESIGN-HIERARCHY-001, principle DS-01)
→ **BUILD** `coder.vite.design-tokens-pure`
- Evidence (AGNOSTIC): `rules[0].description: "Tokens are pure values — no widgets,
  no logic (mirrors principles.DS-01)"`; `layers.tokens.validation: [{type: no_widgets,
  message: "Tokens must not contain widget definitions"}, {type: no_logic, message:
  "Tokens are pure values, no business logic"}]`. Applies to any consumer's token
  files → per-file locatable → mirror.
- Realization: flag, inside a design-system **token** file, (a) a JSX/widget element,
  (b) a `react` import, or (c) a control-flow keyword (`if(`/`for(`/`while(`/`switch(`)
  — the `no_widgets` + `no_logic` checks made concrete for the Vite/TSX stack.

### 2. `coder.design.dependency-flow-tokens-primitives` (DESIGN-HIERARCHY-002, principles DS-02/03/04/05)
→ **BUILD** `coder.vite.design-dependency-flow`
- Evidence (AGNOSTIC): `rules[1].description: "Dependency flow: tokens ← primitives ←
  components ← templates (mirrors principles.DS-05)"`; `dependency_rules.forbidden_edges`
  (`primitives → [components, templates]`, `components → [templates]`, `tokens →
  [primitives, components, templates]`); validation checks `VC-DS-03` ("Primitives
  don't import components") and `VC-DS-04` ("Components don't import templates").
  All are per-import locatable edges → mirror.
- Note: this is the *frontend* sibling of the already-built convex
  `coder.convex.design-hierarchy-import` (`coder.design.hierarchy-import`) — same
  inward-flow principle, **distinct core rule_id** (DESIGN-HIERARCHY-002 ≠
  DESIGN-HIERARCHY-IMPORT-001), and vite had NO hierarchy detector → genuine gap.
- Realization: classify each design-system file to a layer rank
  (tokens 0 < primitives 1 < components 2 < templates 3), resolve each relative
  import to a target layer, flag any edge to a **higher** rank (outward/upward).

### 3. `coder.design.wagons-import-from-design` (DESIGN-HIERARCHY-003, principle DS-06, AP-DS-02)
→ **BUILD** `coder.vite.design-wagons-import`
- Evidence (AGNOSTIC): `rules[2].description: "Wagons import from design_system;
  design_system never imports from wagons (mirrors principles.DS-06)"`;
  `dependency_rules.forbidden_edges: [{from: design_system, to: [wagons], message:
  "Design system cannot import from feature wagons"}]`; `VC-DS-05` (`check_imports:
  {path: "lib/design_system/", forbid_pattern: "package:app/((?!design_system)[^/]+)/"}`,
  `violation: "DS-06"`); `AP-DS-02: "Leaky abstraction … Design system components
  importing from wagons"`. Per-import locatable → mirror.
- Realization: for a file under the design-system root, flag any import that escapes
  the design-system directory into a feature wagon — a relative specifier that climbs
  above the design-system root, or an alias/app import (`@/features/…`, `@/<wagon>/…`,
  `src/<wagon>/…`) that is not itself under design.

These three emits also **subsume** the following agnostic sub-surfaces (no separate
detector needed): principles DS-01/DS-04/DS-05/DS-06; validation checks
VC-DS-03/04/05; anti-pattern AP-DS-02; the `dependency_rules.forbidden_edges` and
`cross_wagon` specs (design→wagon direction).
