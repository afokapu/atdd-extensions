## `backend.convention.yaml` — Backend Component Convention

| core rule_id | alias | classification | mirror rule_id |
|---|---|---|---|
| `coder.backend.code-is-organized` | BOUNDARIES-LAYER-001 | **AGNOSTIC-CONSUMER** | `coder.convex.backend-layer-catalog` |
| `coder.backend.component-file-suffix-matches` | BOUNDARIES-LAYER-002 | **AGNOSTIC-CONSUMER** | `coder.convex.backend-suffix-matches` |
| `coder.backend.imports-respect-dependency-allowed` | BOUNDARIES-LAYER-003 | **AGNOSTIC-CONSUMER** | `coder.convex.backend-dependency-edges` |

**Evidence — all three are consumer-code obligations.** The convention describes a
universal layered-architecture discipline, not toolkit internals:

- `code-is-organized`: *"Backend code is organized into 4 layers: presentation,
  application, integration, domain"* — a property of *any* backend codebase. The
  `backend.layers` catalog enumerates the four layers for every stack; `structure`
  gives per-stack path patterns (`python/…`, `supabase/functions/{wagon}/{feature}/{layer}/`)
  — i.e. it is explicitly realized per-stack. A Convex app is a consumer of this.
- `component-file-suffix-matches`: *"Component file suffix matches its layer's
  component_type catalog (e.g. `*_controller.py` for presentation.controllers)"* — the
  catalog carries an explicit **`typescript`** suffix for every component type
  (`*-controller.ts`, `*-repository.ts`, `*-use-case.ts`, …). Per-stack suffixes ⇒ agnostic.
- `imports-respect-dependency-allowed`: *"Imports respect dependency.allowed_edges
  (presentation→application→domain; integration→application→domain)"*. `dependency.allowed_edges`
  and `forbidden_examples` ("domain → application", "presentation → integration (bypass
  application)") are stack-neutral layering laws; `ci_enforcement.tools` names
  *"eslint/depcruiser or ArchUnit (**per stack**)"* — the detector is per-stack, the
  obligation is universal. → MIRROR.

These three form one cohesive Convex FAMILY validator
`convex_backend_layout_detector` (layout / suffix / dependency-edge checks).

## `dto.convention.yaml` — DTO Convention

| core rule_id | alias | classification | mirror rule_id |
|---|---|---|---|
| `coder.dto.placement` | DTO-PLACEMENT-001 | **AGNOSTIC-CONSUMER** | `coder.convex.dto-placement` |
| `coder.dto.purity` | DTO-PURITY-001 | **AGNOSTIC-CONSUMER** | `coder.convex.dto-purity` |
| `coder.dto.mapper` | DTO-MAPPER-001 | **AGNOSTIC-CONSUMER** | `coder.convex.dto-mapper` |

**Evidence — all three carry an explicit TypeScript realization.** The convention's
`languages.typescript` block gives concrete TS shapes, locations, and naming, so a
Convex/TS app is a first-class consumer:

- `placement`: *"DTOs live in contract_dto modules and are derived from contracts"*;
  `languages.typescript.dto_structure.location_pattern: "ts/contracts/{theme}/{domain}/{resource}.ts"`,
  `naming_convention: "{Resource}DTO"`. Cross-references demand *"DTOs must live in
  neutral contracts/ namespace"* (`enforcement.rules`). → MIRROR: a `*DTO` declaration
  must sit under a `contracts/` module.
- `purity`: `validation.dto_requirements` — *"DTOs MUST be pure data structures with
  NO methods (except serialization)"* and *"DTOs MUST be immutable (… readonly in
  TS)"*. Directly detectable on TS DTO declarations. → MIRROR: a `*DTO` type must be
  readonly data only (no method members, no mutable fields).
- `mapper`: *"Mappers between domain entities and DTOs live in the integration layer"*;
  `mapper_requirements` — *"Mappers MUST live in integration layer, never in domain"*;
  TS mapper location `{wagon}/src/integration/dtoMapping.ts`. → MIRROR: a mapper module
  must sit under an `integration/` layer directory.

These three form one cohesive Convex FAMILY validator `convex_dto_detector`.

## `technology.convention.yaml` — Technology Stack Convention

| core rule_id | alias | classification | mirror? |
|---|---|---|---|
| `coder.technology.new-components-default-to` | COACH-TECH-STACK-001 | **PLANNER-OWNED / SUBSTRATE-SPEC** | NO — why-not |
| `coder.technology.approved-alternatives-are-taken` | COACH-TECH-STACK-002 | **PLANNER-OWNED / SUBSTRATE-SPEC** | NO — why-not |
| `coder.technology.unapproved-technology-choices-require` | COACH-TECH-STACK-003 | **PLANNER-OWNED / SUBSTRATE-SPEC** | NO — why-not |

**Why-not (quoted evidence).** These are stack-**selection governance** rules owned by
the coach/planner layer, not per-file obligations on consumer code:

- The alias namespace is `COACH-TECH-STACK-00x` — a **coach**-owned id family (contrast
  the `BOUNDARIES-LAYER-*` / `DTO-*` ids above, which are structural coder obligations).
- The obligation body is a **project substrate map**, not a universal law: the
  `technology.*` tree hard-codes *this* project's chosen stack — *"Supabase Edge
  Functions (TypeScript on Deno)"*, *"Flutter 3.x"*, *"Riverpod … BLoC/Cubit"*,
  *"Gun.js mesh"*. None of it transfers to a Convex/Vite consumer; the "defaults"
  themselves are the planning decision under governance.
- `coder.technology.new-components-default-to`: *"New components default to the
  `technology.<layer>.default` for their layer; deviations declare an approved
  alternative"* — conformance is defined against a **declared stack map**, i.e. a
  planner/architecture artifact, with no universal detector.
- `coder.technology.unapproved-technology-choices-require`: *"Unapproved technology
  choices require **a SPEC edit** before they can ship"* — explicitly a **spec-layer /
  planner** gate, not a coder detector over consumer source. `notes.deviation_process`
  confirms it: *"Document any alternative choice in the architecture log with rationale
  and expected revisit date."*
- All three are `disposition: documentation-only` **and** have no per-stack detector
  vocabulary (no `typescript`/`per stack` realization hooks — unlike backend/dto).
  Core ships them as documentation, and the substrate they govern is project-specific.

→ DOCUMENT WHY-NOT; do not mirror. (If a future consumer wants an approved-dependency
allowlist detector, that is a *new* planner-configured convention, not a mirror of this
one.)
