# W4 — Commons layering — self-classification

Source of truth (read-only): `/Users/alecfokapu/Github/atdd/main/src/atdd/coder/conventions/commons.convention.yaml`

**Scope:** the three `dependency_rules.rules` registry-walker IDs
(`coder.commons.domain-layer-in-commons`, `coder.commons.application-layer-uses-ports`,
`coder.commons.cross-feature-imports-in`) plus the eleven
`enforcement.validators.specs` (`SPEC-CODER-COMMONS-0001` … `0011`).

**Method:** each row is classified AGNOSTIC-CONSUMER (→ mirror as a per-stack
detector) or ATDD-INTERNAL / SUBSTRATE-SPEC (→ document why-not, quote the proof).
Commons is explicitly cross-stack — the convention states *"Both patterns enforce the
same dependency rules; only the directory structure differs."* — so the dependency-
direction obligations are mirrored into **both** the Convex backend and the Vite
frontend. A consumer Convex/TS app that has a shared/commons module with
`domain/`, `application/`, `integration/` layers is exactly the governed subject.

Grounding: `frg-app/main/apps/game/convex/shared/{domain,contracts}`,
`frg-app/main/apps/game/src/shared/{domain,application,integration,presentation}`
— the real consumer names its module `shared`; the detectors therefore recognize
both the canonical `commons/` and the legacy `shared/` roots.

## Part A — `dependency_rules.rules` (the 3 registry-walker IDs)

| core rule_id | classification | evidence (quoted from core) | realization |
|---|---|---|---|
| `coder.commons.domain-layer-in-commons` | **AGNOSTIC-CONSUMER → MIRROR** | *"Domain layer in commons has no outbound edges (no application/integration imports)"* and `forbidden_edges: {from: domain, to: [application, integration], reason: "Domain must be pure - no external dependencies"}` | `coder.convex.commons-domain-no-outbound` + `coder.vite.commons-domain-no-outbound` |
| `coder.commons.application-layer-uses-ports` | **AGNOSTIC-CONSUMER → MIRROR** | *"Application layer uses ports; integration implements them (no application→integration edges)"* and `forbidden_edges: {from: application, to: [integration], reason: "Application uses ports; integration implements them"}` | `coder.convex.commons-application-no-integration` + `coder.vite.commons-application-no-integration` |
| `coder.commons.cross-feature-imports-in` | **AGNOSTIC-CONSUMER → MIRROR** | *"Cross-feature imports in commons go through the layer above (no peer-to-peer feature edges)"* | `coder.convex.commons-cross-feature-imports-in` + `coder.vite.commons-cross-feature-imports-in` |

**Why agnostic.** All three are worded as generic architectural obligations over a
commons module's `domain/application/integration` layers — *"Dependencies point
inward: integration -> application -> domain"* (`common_principles.dependency
direction: applies_to: "Both stacks"`). Nothing in the text ties them to the ATDD
toolkit's own substrate; any consumer repo with a layered shared module is bound.
The `disposition: documentation-only` in the core `rules` block is an *enforcement
severity*, not a scope limiter — the obligation still governs consumer code, so it is
mirrored (as a `severity: 3` advisory node, preserving the documentation-only intent).

These three form one cohesive **layering family** and are realized as a single
family detector per stack (`convex_commons_layering_detector`,
`vite_commons_layering_detector`) — one detector emitting the group's
`emits_rule_ids`, per the Core family pattern.

## Part B — `enforcement.validators.specs` (SPEC-CODER-COMMONS-0001 … 0011)

These are the **CORE toolkit's own validator specs** — `enforcement.validators.location:
"atdd/coder/validators/test_commons_structure.py"`. Most describe how the ATDD
*reference* multi-stack repo is scaffolded and named, not a portable obligation.
One encodes a genuinely agnostic obligation (domain framework-purity) and IS mirrored.

| spec id | classification | evidence (quoted) | disposition |
|---|---|---|---|
| `SPEC-CODER-COMMONS-0001` "Commons exists in both Python and Frontend" | **SUBSTRATE-SPEC → why-not** | *"validates: Consistent naming across stacks"* — asserts a **cross-stack** monorepo (`both Python and Frontend`); a single-stack Convex-only or Vite-only consumer legitimately has commons in one stack. Not a portable per-file obligation. | not mirrored |
| `SPEC-CODER-COMMONS-0002` "Frontend commons has domain/application/integration layers" | **SUBSTRATE-SPEC → why-not (covered)** | *"validates: Layer-first structure for frontend"* — a scaffolding/existence assertion. The *enforceable* content (the inward dependency direction between those layers) is exactly what the three Part-A rules mirror; the bare "layers exist" scaffolding is not a violation-bearing obligation. | covered by Part A |
| `SPEC-CODER-COMMONS-0003` "Old 'shared' directory does not exist" | **ATDD-INTERNAL → why-not** | naming/migration of the ATDD reference repo. The canonical-name choice is ATDD's: *"Use 'commons' (not 'shared', 'core', 'lib') in all stacks"*. The real consumer `frg-app` **uses `shared/`** — mirroring "shared must not exist" would flag correct consumer code. Arbitrary canonical-name pick, not a safety/architecture obligation. | not mirrored |
| `SPEC-CODER-COMMONS-0004` "Path aliases use @commons not @shared" | **ATDD-INTERNAL → why-not** | *"validates: Correct path alias configuration"* bound to specific ATDD config files (`config_files: web/tsconfig.json, web/vite.config.ts`). Repo-specific naming/config, same arbitrary `@commons` pick as 0003. | not mirrored |
| `SPEC-CODER-COMMONS-0005` "No imports from @shared or ./shared" | **ATDD-INTERNAL → why-not** | *"validates: All imports migrated to @commons"* — the same `commons`-vs-`shared` naming migration; would flag `frg-app`'s real `@shared`/`../shared` imports. | not mirrored |
| `SPEC-CODER-COMMONS-0006` "Frontend domain layer has no framework imports" | **AGNOSTIC-CONSUMER → MIRROR** | *"validates: Domain layer purity (no preact, react, @tanstack)"*; `layers.domain.forbidden_imports.frontend: ["preact","react","@tanstack","@maintain-ux","gsap","@gsap"]`. A generic "the commons domain layer stays framework-agnostic" obligation binding any frontend consumer. | `coder.vite.commons-domain-no-framework-import` |
| `SPEC-CODER-COMMONS-0007` "Frontend commons has proper barrel exports (index.ts)" | **SUBSTRATE-SPEC → why-not** | *"validates: Public API structure"* — a barrel-file *style/scaffolding* preference, not an architecture/safety obligation; absence of an `index.ts` is a stylistic gap, not a dependency violation. | not mirrored |
| `SPEC-CODER-COMMONS-0008` "Python commons has __init__.py files" | **SUBSTRATE-SPEC (python) → why-not** | *"validates: Python package structure"* — Python-package mechanics; no analogue in the Convex/TS target stacks. | not mirrored (out-of-stack) |
| `SPEC-CODER-COMMONS-0009` "Python events feature has internal layer structure" | **SUBSTRATE-SPEC (python) → why-not** | *"validates: Feature-first pattern for complex features"* — asserts the ATDD reference `events/` mini-wagon layout; a repo-specific reference-structure spec. | not mirrored (out-of-stack) |
| `SPEC-CODER-COMMONS-0010` "Python resilience is flat (no internal layers)" | **SUBSTRATE-SPEC (python) → why-not** | *"validates: Flat structure for simple utilities"* — asserts the ATDD reference `resilience/` layout; repo-specific reference structure. | not mirrored (out-of-stack) |
| `SPEC-CODER-COMMONS-0011` "Python domain layer has no framework imports" | **AGNOSTIC obligation, realization out-of-stack → covered** | *"validates: Domain layer purity (no flask, fastapi, django)"* — same agnostic *domain-purity* obligation as 0006 but the **python** realization. Its TS realizations already exist: the frontend build here (`coder.vite.commons-domain-no-framework-import`, 0006) and the pre-existing backend node `coder.convex.domain-no-convex-import` (Convex domain must not import the `convex/*` runtime). | obligation mirrored via 0006 + existing convex node; python detector out of W4 target stacks |

## Build summary (agnostic → realized)

**Backend — `convex_commons_layering_detector`** (family, `official/convex.workspace.runtime`),
conventions in `official/convex.extension.coder`:
- `coder.convex.commons-domain-no-outbound`
- `coder.convex.commons-application-no-integration`
- `coder.convex.commons-cross-feature-imports-in`

**Frontend — `vite_commons_layering_detector`** (family, `official/frontend.workspace.runtime`),
conventions in `official/frontend.extension.vite-coder`:
- `coder.vite.commons-domain-no-outbound`
- `coder.vite.commons-application-no-integration`
- `coder.vite.commons-cross-feature-imports-in`
- `coder.vite.commons-domain-no-framework-import` (0006 / domain framework-purity)

Backend domain framework/runtime purity (0011-analog) is **not** re-built: it is already
owned by the existing `coder.convex.domain-no-convex-import` node.

Astro (`apps/web/src`) is organized as `components/layouts/pages/i18n/styles` with **no**
layered commons module, so it is not a natural target for these layering rules — not mirrored there.
