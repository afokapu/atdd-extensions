# W8 — Coder gap fill (security / boundaries / logging / coverage)

Self-classification of every convention assigned to W8, per the SHARED METHOD in
`docs/MIRROR-GAP-ASSIGNMENT.md`. Each row is one `rule_id` with quoted evidence and
a verdict:

- **AGNOSTIC-CONSUMER → MIRROR** — the obligation applies to any consumer repo; a
  Convex/TS/frontend detector realizes it. (If already realized by a prior wave, the
  row says so and points at the existing node — no duplicate is built.)
- **ATDD-INTERNAL / PLANNER-OWNED / SUBSTRATE-SPEC / CROSS-LANGUAGE-N/A → WHY-NOT** —
  the obligation governs the ATDD toolkit's own substrate, is a planner/plan-graph
  concern, or is structurally un-realizable in a single-language Convex/frontend
  consumer stack. Documented, never silently dropped.

## boundaries.convention.yaml

| rule_id | verdict | evidence |
|---|---|---|
| `coder.boundaries.http-client` | AGNOSTIC — **already mirrored (out of W8 scope)** | Assignment: *"(http-client done)"*. Realized as `coder.vite.boundaries-http-client`. |
| `coder.boundaries.xlang-entity` | **WHY-NOT — CROSS-LANGUAGE-N/A** | `description: "Domain entities must be defined consistently across Python and TypeScript"`; single validator `test_cross_language_consistency::test_entity_classes_exist_across_languages`; `disposition: advisory` with the note *"emits synthetic ``contracts/<name>:0`` locations rather than per-site source lines, so per-site suppression markers don't apply."* |
| `coder.boundaries.xlang-enum` | **WHY-NOT — CROSS-LANGUAGE-N/A** | `description: "Enum values must match across Python and TypeScript domain definitions"`. Same bilingual validator. |
| `coder.boundaries.xlang-naming` | **WHY-NOT — CROSS-LANGUAGE-N/A** | `description: "Naming conventions must be consistent across runtimes for shared concepts"`. Requires ≥2 runtimes to compare. |
| `coder.boundaries.xlang-contract` | **WHY-NOT — CROSS-LANGUAGE-N/A** | `description: "API contracts must be honored consistently across Python and TypeScript"`; `disposition: advisory`, same synthetic-`contracts/<name>:0` note. |

**Why-not rationale (all four xlang rules).** These are *cross-language consistency* checks:
the obligation is that a shared domain concept is defined identically **across Python AND
TypeScript**. Unlike `http-client` (a single-stack obligation with a per-stack detector, hence
mirror-able), an xlang rule is inherently *bi-lingual* — it compares two runtimes and emits a
synthetic `contracts/<name>:0` location, not a per-file source site. The mirror target stacks
(`convex`/`vite`/`astro`, all in `frg-app`: `apps/game/convex/`, `apps/game/src/`, `apps/web/src/`)
are **single-language TypeScript** — there is no Python side to be consistent *with*, so the
detector has nothing to compare and no dirty fixture can make it fire. This is not ATDD-internal;
it is structurally un-realizable in a mono-language consumer stack. Documented, not dropped.

**Namespacing / interaction prose (Sections 1–2 of the convention).** The bulk of
`boundaries.convention.yaml` (`namespacing.*`, `import_patterns.forbidden_patterns`,
`test_configuration.pytest.pythonpath`, `conftest_prohibition`, `composition_roots.*`,
`station_master_pattern`) carries **no registry-walkable `rule_id`** and is
**Python-package-substrate guidance**: `pattern: "from domain.X import Y"`,
`setting: "pythonpath = [\".\"]"`, `sys.path.insert(0, str(src_path))`,
`file: "python/{wagon}/{feature}/composition.py"`. It governs Python module resolution and
`pyproject.toml`/`conftest.py`; a Convex/TS consumer has ES-module resolution and no
`sys.path`/`pythonpath`. **WHY-NOT — SUBSTRATE-SPEC (Python-only), no rule_id to mirror.**

## coverage.convention.yaml

| rule_id | verdict | evidence |
|---|---|---|
| `coder.coverage.every-feature-must-have` (COVERAGE-CODE-4.1) | **WHY-NOT — PLANNER-OWNED (hierarchy-graph)** | `description: "Every feature must have corresponding code implementation"`, `requirement: "Each feature's components must map to code paths"`, `validator: test_hierarchy_coverage::test_all_features_have_implementations`; convention header: *"Section 4 of ATDD Hierarchy Coverage Spec v0.1"*; `rollout.phase: FULL_ENFORCEMENT` — *"Section 4 validators become strict in Phase 3 (after planner+tester)"*. |
| `coder.coverage.every-implementation-must-have` (COVERAGE-CODE-4.2) | **WHY-NOT — PLANNER-OWNED (hierarchy-graph)** | `description: "Every implementation must have at least one linked test"`, `validator: test_hierarchy_coverage::test_all_implementations_have_tests`; `coverage_graph.levels` walks `Feature → Implementation → tested_by`. |

**Why-not rationale (both coverage rules).** These are **hierarchy-graph traceability**
validators, not per-file source detectors. The "feature" and "implementation" nodes come from
the ATDD **plan/hierarchy registry** (feature URNs, the implementation registry, the tested-by
graph) — `exception_handling.allow_lists.features_without_implementation` is a *plan-config*
key (`config_path: coverage.exceptions.features_without_implementation`). A violation is
"feature *X* (declared in the plan) has no code" / "implementation *Y* has no linked test" —
which has **no source line/col** and cannot be produced by the per-file
`ATDD_SCAN_ROOTS → {rule_id,file,line,col}` detector contract, since the detector only sees
scanned source, not the plan graph. Enforcement is explicitly sequenced *after* planner+tester
(`Phase 3`). This is a planner/plan-layer concern owned by the hierarchy-coverage graph, not an
agnostic per-file consumer pattern. Documented, not dropped.
